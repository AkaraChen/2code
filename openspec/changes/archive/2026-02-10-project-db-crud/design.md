## Context

The backend is a Tauri 2 Rust app. The existing PTY module uses an in-memory `Arc<Mutex<HashMap>>` registry managed via `tauri::State`. There is no database, ORM, or persistence layer. Commands follow a consistent pattern: `#[tauri::command]` functions receiving `State<'_, T>`, returning `Result<T, String>`.

We need to add a persistence layer for Project records. The new `project` module should follow the same structural conventions as the `pty` module (separate files for models, commands) but backed by SQLite instead of in-memory state.

## Goals / Non-Goals

**Goals:**
- Embed SQLite via Diesel ORM for persistent Project storage.
- Provide full CRUD Tauri commands for projects.
- Support two project creation modes: temporary (auto `/tmp` + `git init`) and from existing folder.
- Auto-run migrations on app startup.
- Follow the same module conventions established by the PTY module.

**Non-Goals:**
- Frontend UI for project management (separate change).
- Filesystem watching or syncing project folder contents.
- Project-scoped PTY sessions (future work).
- Multi-user or remote database support.

## Decisions

### 1. ORM: Diesel

**Choice**: Diesel with `sqlite` feature and embedded migrations (`diesel_migrations`).

**Rationale**: Diesel is the most mature Rust ORM with strong compile-time query validation and a powerful schema DSL. Its synchronous API is fine for this use case — SQLite operations are fast local I/O, and Tauri commands can be async while running Diesel calls on a blocking thread via `tokio::task::spawn_blocking` (same pattern already used for PTY I/O). The `diesel_migrations` crate embeds SQL migrations in the binary for auto-run at startup.

**Alternatives considered**:
- **SeaORM**: Async-native but heavier dependency tree, more boilerplate for entity definitions. Less mature ecosystem.
- **rusqlite (raw)**: Lightweight but no migration system, no query builder, manual SQL strings. More error-prone as the schema grows.

### 2. Database location: Tauri app data directory

**Choice**: Store `app.db` in the Tauri-resolved app data directory (`AppHandle::path().app_data_dir()`).

**Rationale**: Cross-platform, respects OS conventions (`~/Library/Application Support/` on macOS, `%APPDATA%` on Windows). Tauri provides this path reliably. The `dirs` crate is already a dependency but Tauri's resolver is more idiomatic.

### 3. State management: `DbPool` in Tauri managed state

**Choice**: Wrap a Diesel `SqliteConnection` in an `Arc<Mutex<>>` (type-aliased as `DbPool`) and pass it via `.manage()`, just like the PTY registry.

**Rationale**: Consistent with the existing `Arc<Mutex<HashMap>>` pattern used for PTY sessions. Diesel's `SqliteConnection` is not `Sync`, so it needs a mutex. For a single-user desktop app, a single connection behind a mutex is sufficient — no connection pool needed.

### 4. ID generation: UUID v4 string

**Choice**: Use `uuid::Uuid::new_v4().to_string()` as the primary key, stored as TEXT in SQLite.

**Rationale**: Consistent with PTY session IDs. Avoids auto-increment integer IDs that leak ordering info. The `uuid` crate is already a dependency.

### 5. Module structure

```
src-tauri/src/
├── project/
│   ├── mod.rs          # module exports
│   ├── models.rs       # Diesel Queryable/Insertable structs
│   ├── schema.rs       # Diesel table! macro (auto-generated by diesel CLI)
│   └── commands.rs     # Tauri command handlers
├── db.rs               # DB init helper (connect + run embedded migrations)
├── pty/
│   └── ...
└── lib.rs              # registers project commands + DB state
```

Migrations live in `src-tauri/migrations/` as SQL files (managed by `diesel migration generate`). They are embedded into the binary via `diesel_migrations::embed_migrations!`.

Follows the PTY module's separation of concerns: models in one file, commands in another.

### 6. Error handling: consistent `Result<T, String>`

**Choice**: All project commands return `Result<T, String>`, mapping ORM errors via `.map_err(|e| e.to_string())`.

**Rationale**: Matches the PTY module convention exactly. Keeps the IPC contract simple.

### 7. Temporary project creation: `/tmp/<uuid>/`

**Choice**: `create_project_temporary` generates a UUID, creates `/tmp/<uuid>/`, runs `git init` via `std::process::Command`, then inserts the record.

**Rationale**: `/tmp` is ephemeral by design — appropriate for scratch/temporary projects. The UUID directory name avoids collisions. `git init` is a simple subprocess call; no git library dependency needed.

## Risks / Trade-offs

- **Diesel sync API**: Diesel is synchronous; SQLite calls block the thread. → Use `tokio::task::spawn_blocking` in async Tauri commands to avoid blocking the async runtime. SQLite local I/O is fast so this is negligible.
- **Migration on startup**: If a migration fails, the app won't start. → Migrations are simple (single CREATE TABLE) and tested. Log errors clearly.
- **`/tmp` cleanup**: OS may clean `/tmp` contents. Temporary projects could lose their folders. → This is expected behavior for "temporary" — document it. The DB record persists so the user sees the project still listed.
- **No folder validation on startup**: A project's folder may be deleted externally. → Out of scope for this change. Future work can add a `valid` flag or check on access.
